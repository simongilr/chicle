# Chicle Platform Architecture

Chicle follows an **evolutionary modular platform architecture**.

The project starts as a simple local MVP, but it is intentionally shaped as a configurable product factory: one shared
core, multiple business templates, dynamic runtime objects and several possible delivery targets.

## Definition

Chicle is a modular and evolutionary platform for building configurable digital products from templates. It can power a
web app, mobile app, desktop app or administrative system from a common core, dynamic runtime, declarative services,
workflows, records, actions and tenant-scoped configuration.

Technical definition:

> Chicle is a metadata-driven modular monolith, prepared for event-driven, multi-tenant and multi-target execution.

## What Chicle Is Not

Chicle is not a single closed application.

Chicle is not only a CMS.

Chicle is not only a low-code form builder.

Chicle is not a microservice platform in V1.

Chicle is a platform base that can start small and grow into a broader product factory without changing its core
contract.

## Architecture Mix

Chicle combines several architectural ideas:

| Pattern | Role In Chicle |
| --- | --- |
| Modular Monolith | The first deployable shape stays simple: API, app and database. Modules are separated by domain boundaries. |
| Metadata-Driven Architecture | Forms, services, flows, menus, actions and future screens are stored as declarative configuration. |
| Template-Based Product Engine | Business verticals are installed as templates/seeds and do not leak product rules into core. |
| Low-Code/Internal Platform | Administrators can configure behavior through visual designers and JSON contracts. |
| Event-Driven Ready Architecture | Records, outbox events, flow jobs and future workers prepare async execution without forcing it in V1. |
| Multi-Target Application Factory | The same contract can render for web, mobile, desktop or admin surfaces through presentation adapters. |

## Evolution Path

The first production shape should stay intentionally simple:

```txt
api
db
app
```

Chicle is prepared to evolve toward:

```txt
api
api-worker
db
backup
event engine
AI assistant
template installer
web target
mobile target
desktop target
admin target
```

This is evolutionary architecture: the early system avoids unnecessary infrastructure, but the core contracts leave room
for workers, event processing, AI-assisted authoring, template installation and multi-target rendering later.

## Core Principles

### 1. Generic Core, Business Templates

The core contains generic runtime capability. Business concepts live in templates.

Core can know:

- dynamic services
- dynamic forms
- flows
- records
- actions
- files
- users, roles and permissions
- tenant settings

Templates can know:

- logistics concepts
- real-estate concepts
- ticketing concepts
- services business concepts
- game concepts
- any tenant vertical vocabulary

### 2. Declarative Runtime

Chicle stores behavior as validated JSON, not arbitrary code.

Allowed:

- service definitions
- form schemas
- flow definitions
- action bindings
- presentation profiles
- role resource policies

Not allowed:

- arbitrary JavaScript from database configuration
- raw SQL generated by users or AI
- secrets inside JSON contracts
- tenant data leaking into another tenant's context

### 3. One Contract, Multiple Targets

The same business object should be usable by multiple targets:

- web admin
- mobile app
- desktop app
- future public portal
- future embedded workflow

Presentation uses adapters and themes. Stored configuration should remain library-neutral.

### 4. Visual And JSON Authoring Are Equal

Humans use guided visual designers.

Advanced users and future AI assistants can use JSON directly.

Both paths write the same contracts and go through the same backend validation.

### 5. Event-Driven Ready, Not Event-Driven Forced

Chicle can start synchronously and locally. As needs grow, records, events, flow jobs, outbox, workers and live updates
can activate without changing the object contracts.

### 6. AI-Assisted, Backend-Validated

AI can help generate configuration. It cannot bypass Chicle validation.

The future AI Manager uses RAG and controlled tools to propose drafts, but the API remains the execution authority.

## Current Architectural Position

Current Chicle already contains the foundation:

- NestJS modular API.
- Angular/Ionic/PrimeNG frontend.
- Tenant-scoped MariaDB persistence.
- Dynamic Services stored and executed from DB.
- Dynamic Forms with JSON authoring and Formly runtime bridge.
- Flow Engine with draft, preview, publish, jobs, outbox and lifecycle pieces.
- Shared frontend component catalog.
- JSON-only authoring endpoints for future AI.
- RAG architecture for versioned knowledge packs.

This means Chicle is already beyond a normal admin application. It is becoming an internal product platform.

## Product Factory Interpretation

A product factory in Chicle means:

1. Choose or install a template.
2. Configure tenant-specific parameters.
3. Create or adjust dynamic services.
4. Create forms and screens.
5. Compose flows and actions.
6. Publish versions.
7. Render the experience in web, mobile, desktop or admin targets.
8. Iterate safely through versioned contracts.

The same core can support different verticals without copying the application.
